# Tippa Mobile ‚Äî Auth Migration: Clerk ‚Üí JWT

## What This Document Is

This is a snapshot of the **current** auth architecture (Clerk-based) before migrating to a phone + access code + JWT system. It documents every file that needs to change, what it currently does, and what it must become.

---

## Current Architecture (Clerk-based)

### How auth works today

1. Guard opens the app
2. Clerk SDK initialises via `ClerkProvider` in `app/_layout.tsx`
3. `ClerkGate` component waits for `useAuth().isLoaded` before rendering anything
4. `app/index.tsx` checks `isSignedIn` and `user` from Clerk
5. If signed in, calls `GET /api/mobile/check-link/:clerkUserId` to see if the Clerk user is linked to a CarGuard record
6. If linked, saves guard data to AsyncStorage and redirects to `/(tabs)`
7. If not linked, redirects to `/(auth)/link-account` where they enter a 6-digit access code to link their Clerk account to their guard profile
8. Every subsequent API call passes `user.id` (the Clerk user ID) as a URL param

```

  ‚Üí ClerkProvider loads SDK
  ‚Üí useAuth().isLoaded = true
```

## New Architecture (JWT-based)

### How auth will work after migration

1. Guard opens the app
2. `AuthContext` checks AsyncStorage for a saved JWT
3. If JWT exists and is valid ‚Üí load guard data ‚Üí go to `/(tabs)`
4. If JWT missing or expired ‚Üí go to `/(auth)/sign-in`
5. Sign-in screen: enter phone number ‚Üí enter 6-digit access code
6. `POST /api/auth/login { phoneNumber, accessCode }` on Tippa backend
7. Backend normalises phone, finds CarGuard, validates access code, returns JWT + guard data
8. JWT stored in SecureStore, guard data in AsyncStorage
9. All subsequent API calls send `Authorization: Bearer <token>` header
10. Backend middleware verifies JWT and attaches guard to request
11. No clerkUserId anywhere in the mobile flow

| `app/(tabs)/payouts.tsx` | Replace `user.id` with guard id from AuthContext |
| `contexts/GuardContext.tsx` | Replace `clerkUserId` param with JWT-authenticated fetch |
| `contexts/AuthContext.tsx` | **New file** ‚Äî JWT storage, login, logout, session check |

|------|--------|

### New JWT token payload

```json
{
  "guardId": "698b232583112442dab4ddef",
  "guardPublicId": "88018372-3244-43ef-8c49-8b9035278653",
  "phone": "0821234567",
  "iat": 1713348000,
  "exp": 1715940000
}
```

### New login endpoint contract

```
POST /api/auth/login
Body: { phoneNumber: string, accessCode: string }

Success 200:
{
  success: true,
  token: "<jwt>",
  guard: {
    id: string,
    guardId: string,
    name: string,
    surname: string,
    fullName: string,
    phone: string,
    balance: number,
    status: string,
    qrCode: string | null,
    qrCodeUrl: string | null
  }
}

Error 401:
{ success: false, error: "Invalid phone number or access code" }

Error 401:
{ success: false, error: "Access code has expired. Please request a new one from your manager." }
```

### Protected route middleware contract

```typescript
// Applied to all /api/mobile/* routes
// Reads: Authorization: Bearer <token>
// Attaches: req.guard = { id, guardId, phone }
// Returns 401 if missing or invalid token
```

## What Is NOT Changing

- `web/server.js` ‚Äî untouched. Clerk stays for admin login.
- `backend/src/routes/clerk-webhook.ts` ‚Äî untouched. Clerk webhooks continue syncing web users.
- Admin portal ‚Äî no changes.
- The access code generation flow on the web portal ‚Äî unchanged. Admins still generate codes. Guards now use them to log in directly instead of to link a Clerk account.
- The `CarGuard.accessCode` and `CarGuard.accessCodeExpiry` fields ‚Äî still used, same as before.
- The `CarGuard.clerkUserId` field ‚Äî can be kept as-is for historical data, just not used by mobile anymore.

## Phone Number Normalisation (backend)

Guards are registered with various phone formats. The login endpoint must accept all and normalise to a standard format before looking up the DB.

```
0821234567    ‚Üí 0821234567  (stored format)
+27821234567  ‚Üí 0821234567
27821234567   ‚Üí 0821234567
082 123 4567  ‚Üí 0821234567
```

Lookup strategy: normalise the input, then query `WHERE phone = normalised`.

## Access Code Behaviour

- 6-digit numeric code
- Generated by admin on web portal (already implemented)
- Stored on `CarGuard.accessCode`, expires at `CarGuard.accessCodeExpiry`
- After successful login: **clear the code** (`accessCode = null`, `accessCodeExpiry = null`) so it cannot be reused
- If expired: return error telling guard to request a new code from their manager

## Session Management

- JWT stored in `expo-secure-store` (encrypted, not plain AsyncStorage)
- Guard profile data cached in AsyncStorage for offline reads
- On app start: read JWT from SecureStore ‚Üí verify expiry ‚Üí if valid, load guard data
- Token refresh: call `POST /api/auth/refresh` with current token ‚Üí get new token with 30-day expiry
- Logout: clear SecureStore JWT + AsyncStorage guard data ‚Üí redirect to sign-in
